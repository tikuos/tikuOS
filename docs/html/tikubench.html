TikuBench Testing

TikuBench is a Python-based hardware-in-the-loop test framework for TikuOS. It automates the full cycle: select a test category, modify the firmware configuration, build, flash, monitor UART output, parse results, and display a summary.

Workflow

  1. Select category — choose which test module to run (e.g., matrix, process, timer).
  2. Modify config — TikuBench updates tests/tiku_test_config.h to enable the selected category and disable all others.
  3. Build — runs make MCU=msp430fr5969 to compile the firmware with the selected tests.
  4. Flash — loads the binary onto the connected MSP430 LaunchPad via mspdebug.
  5. Monitor — opens the UART backchannel and captures output.
  6. Parse — interprets the test protocol markers in the output.
  7. Display — shows a color-coded summary of pass/fail results.

Command-Line Arguments

Argument Default Description
--mcu msp430fr5969 Target MCU for build and flash
--port /dev/ttyACM0 Serial port for UART communication
--baud 9600 UART baud rate
--category (interactive) Test category to run (skips interactive selection)
--list — List all available test categories and exit
--no-interactive — Run without interactive prompts (requires --category)
--timeout 30 Seconds to wait for test output before timing out
--echo — Enter echo mode for UART loopback testing

Running Tests

Interactive Mode

python3 tikubench/runner.py

Presents a numbered menu of all available test categories. Select one and TikuBench handles the rest.

Non-Interactive Mode

# Run matrix tests on FR5969
python3 tikubench/runner.py --category matrix --no-interactive

# Run process tests on a specific port
python3 tikubench/runner.py --mcu msp430fr5969 --port /dev/ttyACM1 \
    --category process --no-interactive

# List all available categories
python3 tikubench/runner.py --list

Echo Mode

Echo mode is useful for UART loopback testing. TikuBench sends data to the device and verifies it is echoed back correctly:

python3 tikubench/runner.py --echo

Test Protocol

Tests communicate results over UART using a simple text-based protocol. Each marker must appear on its own line:

Marker Meaning
[TS:BEGIN] Test suite has started
[TG:BEGIN] A test group (category) has started
[T:P:N] Test named N passed
[T:F:N] Test named N failed
[TG:END] Current test group has finished
[TS:END] Entire test suite has finished

Example UART output from a test run:

[TS:BEGIN]
[TG:BEGIN]
[T:P:arena_create_basic]
[T:P:arena_alloc_single]
[T:P:arena_alloc_until_full]
[T:F:arena_alignment_edge]
[TG:END]
[TS:END]

Adding a New Test Category

  1. Create the test file — add a new .c file in tests/ following the naming convention test_<category>.c.
  2. Add configuration flags — define TEST_<CATEGORY> and sub-test flags in tests/tiku_test_config.h.
  3. Register with the runner — add a call to your test function in tests/test_runner.c guarded by #if TEST_<CATEGORY>.
  4. Use the test protocol — emit [TG:BEGIN], [T:P:name] or [T:F:name], and [TG:END] markers in your test output.
  5. Add to Makefile — include the new .c file in the SRCS list for the test build.
  6. Register in TikuBench — the Python runner auto-discovers categories from the config header, so no Python changes are needed.

Bugs Found

TikuBench has been instrumental in discovering real bugs in the TikuOS kernel during development:

  • Premature GIE in htimer init — the hardware timer initialization enabled global interrupts too early, before the process system was ready to handle timer ticks.
  • Premature GIE in boot — a similar issue where interrupts were enabled during the boot sequence before all subsystems were initialized.
  • Missing EXITED broadcast — when a process exited, the TIKU_EVENT_EXITED broadcast was not being sent, causing the timer process to leak timers belonging to dead processes.
Note: All three bugs were caught by automated TikuBench tests running on real hardware, not by code inspection. This demonstrates the value of hardware-in-the-loop testing for embedded systems.
