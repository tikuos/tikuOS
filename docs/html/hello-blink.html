Hello Blink

The classic embedded "Hello World" — blink an LED using TikuOS processes and timers. This walkthrough explains every line of code.

The Complete Program

#include "tiku.h"

/* 1. Declare the process */
TIKU_PROCESS(blink_process, "Blink");

/* 2. Register it for autostart */
TIKU_AUTOSTART_PROCESSES(&blink_process);

/* 3. Define the process thread */
TIKU_PROCESS_THREAD(blink_process, ev, data)
{
    static struct tiku_timer timer;

    TIKU_PROCESS_BEGIN();

    /* Configure LED pin as output */
    TIKU_BOARD_LED1_DIR |= TIKU_BOARD_LED1_PIN;

    /* Set a periodic timer: 1 second interval */
    tiku_timer_set_event(&timer, TIKU_CLOCK_SECOND,
                         &blink_process);

    while (1) {
        /* Wait until the timer fires */
        TIKU_PROCESS_WAIT_EVENT_UNTIL(ev == TIKU_EVENT_TIMER);

        /* Toggle the LED */
        TIKU_BOARD_LED1_OUT ^= TIKU_BOARD_LED1_PIN;

        /* Reset the timer (drift-free) */
        tiku_timer_reset(&timer);
    }

    TIKU_PROCESS_END();
}

Line-by-Line Explanation

#include "tiku.h"

The master TikuOS header. It pulls in all kernel subsystems (processes, timers, events), the device and board abstraction headers, and platform configuration. Every TikuOS application includes this single header.

TIKU_PROCESS(blink_process, "Blink")

This macro declares a process. It expands to a static struct tiku_process definition with the given variable name and a human-readable string label. The struct contains:

  • next — linked-list pointer for the process queue
  • name — the string "Blink", used by ps and debug output
  • thread — function pointer to the protothread entry point
  • pt — the protothread continuation state (an unsigned short line number)
  • is_running — flag indicating the process is active

The expansion looks roughly like:

static int blink_process_thread(struct tiku_process *p,
                                tiku_event_t ev,
                                void *data);
struct tiku_process blink_process = {
    NULL, "Blink", blink_process_thread, {0}, 0
};

TIKU_AUTOSTART_PROCESSES(&blink_process)

Registers one or more processes to be started automatically during boot. The macro creates a null-terminated array of process pointers. During main(), the kernel calls tiku_autostart_start() which iterates this array, starts each process, and delivers an TIKU_EVENT_INIT event to each one.

You can list multiple processes separated by commas:

TIKU_AUTOSTART_PROCESSES(&blink_process, &sensor_process, &logger_process);

TIKU_PROCESS_THREAD(blink_process, ev, data)

Defines the protothread entry point for the process. This macro expands to a function signature:

static int blink_process_thread(struct tiku_process *process,
                                tiku_event_t ev,
                                void *data)

The scheduler calls this function every time an event is dispatched to the process. The parameters are:

  • ev — the event type (e.g., TIKU_EVENT_INIT, TIKU_EVENT_TIMER)
  • data — an optional data pointer associated with the event

static struct tiku_timer timer;

Important: Local variables in a protothread that must survive across yield points must be declared static. Protothreads are stackless — each call to the thread function re-enters at the saved continuation point, so ordinary stack variables lose their values.

TIKU_PROCESS_BEGIN()

Marks the start of the protothread body. This macro expands to a switch statement that jumps to the saved continuation line number. Every process thread must have a matching TIKU_PROCESS_BEGIN() and TIKU_PROCESS_END() pair. Code before BEGIN runs on every invocation; code after BEGIN resumes from where the thread last yielded.

tiku_timer_set_event(&timer, TIKU_CLOCK_SECOND, &blink_process)

Arms a software event timer. The parameters are:

  • &timer — pointer to the timer struct
  • TIKU_CLOCK_SECOND — the interval in clock ticks (128 ticks = 1 second)
  • &blink_process — the process that will receive the TIKU_EVENT_TIMER event when the timer expires

When the timer fires, the kernel posts TIKU_EVENT_TIMER to the target process's event queue.

TIKU_PROCESS_WAIT_EVENT_UNTIL(ev == TIKU_EVENT_TIMER)

Yields the protothread and waits until the condition is true. The thread saves its current line number and returns control to the scheduler. On the next event dispatch, the thread resumes here and checks the condition. If the event is not a timer event (e.g., it could be a POLL), the thread yields again.

A simpler variant, TIKU_PROCESS_WAIT_EVENT(), yields and resumes on any event without checking a condition.

TIKU_BOARD_LED1_OUT ^= TIKU_BOARD_LED1_PIN

Toggles the LED using an XOR on the GPIO output register. The TIKU_BOARD_LED1_OUT and TIKU_BOARD_LED1_PIN macros are defined in the board header (tiku_board_fr5969_launchpad.h) and map to the correct MSP430 port registers for LED1 (P1.0 on the FR5969 LaunchPad).

tiku_timer_reset(&timer)

Resets the timer relative to its original expiration time, not the current time. This provides drift-free periodic timing: even if the process was delayed slightly in handling the event, the next expiration is calculated from when the timer should have fired, not when it actually was processed.

Compare with tiku_timer_restart(&timer), which resets relative to now — useful for timeout patterns but not for periodic timing.

TIKU_PROCESS_END()

Marks the end of the protothread body. If execution reaches this point (i.e., the thread does not loop forever), the process exits and is removed from the scheduler. In this example, the while (1) loop means we never reach PROCESS_END, which is the normal pattern for long-running processes.

Build and Flash

Save the code above as examples/blink/blink.c, then build and flash:

# Build with the blink example
make MCU=msp430fr5969

# Flash onto the LaunchPad
make flash

You should see the red LED on the LaunchPad toggling once per second.

What to Try Next

  • Change TIKU_CLOCK_SECOND to TIKU_CLOCK_SECOND / 4 for a faster blink (4 Hz).
  • Toggle both LEDs by adding TIKU_BOARD_LED2_OUT ^= TIKU_BOARD_LED2_PIN.
  • Add a second process that blinks at a different rate — see how cooperative multitasking works in practice.
  • Read the Processes page to understand the full process lifecycle.
