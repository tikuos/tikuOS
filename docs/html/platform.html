Platform Abstraction

TikuOS uses a two-level abstraction for hardware support: device headers define silicon-level constants (memory sizes, peripheral availability), and board headers define PCB-level assignments (LED pins, button pins, UART routing). A selector header routes to the correct pair based on a single TIKU_DEVICE_* define.

Two-Level Abstraction

Device Headers (Silicon)

Located in arch/msp430/devices/. Each device header defines the hardware capabilities of a specific MSP430 variant:

  • GPIO port availability (TIKU_DEVICE_HAS_PORT1 through TIKU_DEVICE_HAS_PORTJ)
  • Crystal pin routing (LFXT, HFXT)
  • Clock capabilities (max stable MHz, CS key protection)
  • Memory sizes (FRAM, SRAM, address ranges)
  • MPU segment boundaries
  • Peripheral availability (eUSCI, ADC)

Board Headers (PCB)

Located in arch/msp430/boards/. Each board header defines the pin assignments for a specific development board or custom PCB:

  • LED macros (TIKU_BOARD_LED1_INIT, _ON, _OFF, _TOGGLE)
  • Button macros (TIKU_BOARD_BTN1_INIT, _PRESSED)
  • UART pin configuration and baud-rate constants
  • I2C pin configuration and clock prescalers
  • SPI pin configuration
  • 1-Wire pin assignments

Device Header Example

The FR5969 device header (arch/msp430/devices/tiku_device_fr5969.h) defines constants like:

#define TIKU_DEVICE_NAME            "MSP430FR5969"

/* Memory sizes */
#define TIKU_DEVICE_FRAM_SIZE       (64 * 1024UL)   /* 64 KB FRAM */
#define TIKU_DEVICE_RAM_SIZE        (2 * 1024UL)    /* 2 KB SRAM */
#define TIKU_DEVICE_RAM_START       0x1C00U

/* FRAM address range */
#define TIKU_DEVICE_FRAM_START      0x4400U
#define TIKU_DEVICE_FRAM_END        0xFFFFU

/* MPU segment boundaries */
#define TIKU_DEVICE_MPU_SEG2_START  0x8000U
#define TIKU_DEVICE_MPU_SEG3_START  0xC000U

/* Clock */
#define TIKU_DEVICE_HAS_LFXT        1
#define TIKU_DEVICE_HAS_HFXT        1
#define TIKU_DEVICE_MAX_STABLE_MHZ  8

Board Header Example

The FR5969 LaunchPad board header (arch/msp430/boards/tiku_board_fr5969_launchpad.h) maps pins:

#define TIKU_BOARD_NAME             "MSP430FR5969 LaunchPad"

/* LED1 (Red) - P4.6 */
#define TIKU_BOARD_LED1_INIT()      do { P4DIR |= BIT6; P4OUT &= ~BIT6; } while(0)
#define TIKU_BOARD_LED1_TOGGLE()    do { P4OUT ^= BIT6; } while(0)

/* LED2 (Green) - P1.0 */
#define TIKU_BOARD_LED2_INIT()      do { P1DIR |= BIT0; P1OUT &= ~BIT0; } while(0)
#define TIKU_BOARD_LED2_TOGGLE()    do { P1OUT ^= BIT0; } while(0)

/* Backchannel UART - TXD P2.0, RXD P2.1 */
#define TIKU_BOARD_UART_BRW         52    /* 9600 baud from 8 MHz */

/* Button S1 - P4.5 (Active low) */
#define TIKU_BOARD_BTN1_INIT()      do { P4DIR &= ~BIT5; P4REN |= BIT5; \
                                         P4OUT |= BIT5; } while(0)
#define TIKU_BOARD_BTN1_PRESSED()   (!(P4IN & BIT5))

tiku_device_select.h

The device selector header (arch/msp430/tiku_device_select.h) routes to the correct device and board headers based on a single define:

/* Auto-detect from TI compiler defines */
#if defined(__MSP430FR5969__) && !defined(TIKU_DEVICE_MSP430FR5969)
#define TIKU_DEVICE_MSP430FR5969 1
#elif defined(__MSP430FR5994__) && !defined(TIKU_DEVICE_MSP430FR5994)
#define TIKU_DEVICE_MSP430FR5994 1
#elif defined(__MSP430FR2433__) && !defined(TIKU_DEVICE_MSP430FR2433)
#define TIKU_DEVICE_MSP430FR2433 1
#endif

/* Route to correct headers */
#if defined(TIKU_DEVICE_MSP430FR5969)
#include <arch/msp430/devices/tiku_device_fr5969.h>
#include <arch/msp430/boards/tiku_board_fr5969_launchpad.h>
#elif defined(TIKU_DEVICE_MSP430FR5994)
#include <arch/msp430/devices/tiku_device_fr5994.h>
#include <arch/msp430/boards/tiku_board_fr5994_launchpad.h>
#elif defined(TIKU_DEVICE_MSP430FR2433)
#include <arch/msp430/devices/tiku_device_fr2433.h>
#include <arch/msp430/boards/tiku_board_fr2433_launchpad.h>
#else
#error "No TikuOS device selected."
#endif

HAL Layer

Between the kernel and the architecture-specific code sits a HAL (Hardware Abstraction Layer). Each HAL header declares the interface that the arch code must implement:

HAL Header Purpose
hal/tiku_cpu.h CPU control: boot init, frequency config, atomic sections, interrupt enable/disable
hal/tiku_clock_hal.h System clock: init, time query, tick configuration
hal/tiku_watchdog_hal.h Watchdog timer: stop, start, configure
hal/tiku_mem_hal.h Memory: alignment constants, size types, secure wipe, NVM copy
hal/tiku_mpu_hal.h MPU: segment config, lock/unlock, violation handling

All arch .c files use generic TIKU_DEVICE_* and TIKU_BOARD_* macros rather than directly accessing hardware registers. This keeps the arch code device-agnostic within the MSP430 family.

Adding a New Variant

Adding support for a new MSP430 variant requires four steps:

  1. Device header — create arch/msp430/devices/tiku_device_frXXXX.h with silicon constants (memory sizes, crystal pins, peripheral availability). Use tiku_device_fr5969.h as a template.
  2. Board header — create arch/msp430/boards/tiku_board_frXXXX_launchpad.h with GPIO pin assignments (LEDs, buttons, UART). Use tiku_board_fr5969_launchpad.h as a template.
  3. Selector clause — add an #elif in arch/msp430/tiku_device_select.h to route the new TIKU_DEVICE_MSP430FRXXXX define to the new headers.
  4. Linker script — provide a linker command file (lnk_msp430frXXXX.cmd) that defines the memory layout for the new device.
Note: No changes to the kernel or arch .c files should be necessary. The entire adaptation happens through the device/board header macros and the linker script.

Build System

The build system uses the MCU variable to select the target:

make MCU=msp430fr5969    # Build for FR5969
make MCU=msp430fr5994    # Build for FR5994
make MCU=msp430fr2433    # Build for FR2433

The Makefile auto-detects the toolchain (TI GCC or CCS) and sets compiler flags accordingly. Source files are collected via SRCS wildcard patterns from kernel/, arch/, boot/, and optionally tests/, examples/, or apps/.
