Scheduler

The TikuOS scheduler is the central event loop. It drains the event queue, checks for expired timers, and enters a low-power idle state when no work is pending. The entire scheduler lives in a single file and exposes a small API that main.c calls at startup.

What the Scheduler Does

On every iteration the scheduler performs three steps:

  1. Drain events — dispatch all pending events from the process event queue by repeatedly calling tiku_sched_run_once().
  2. Check timers — the timer process is polled by the clock ISR; expired timers post events that get drained in step 1.
  3. Enter idle — if no events remain and no timers are due, call the idle hook inside an atomic section so the platform can enter a low-power mode without missing interrupts.

Startup Sequence

The boot code in main.c calls exactly two scheduler functions:

tiku_sched_init();   /* initialize all subsystems */
tiku_sched_loop();   /* run forever */

tiku_sched_init()

Initializes the scheduler and all managed subsystems. The order matters:

  1. Process system — tiku_process_init() resets the process list and event queue.
  2. Hardware timer — tiku_htimer_init() sets up the platform timer peripheral (Timer_A on MSP430).
  3. Software timers — tiku_timer_init() starts the timer management process, which requires the process system to already be running.
void tiku_sched_init(void)
{
    sched_state = TIKU_SCHED_RUNNING;
    idle_hook = (tiku_sched_idle_hook_t)0;

    tiku_process_init();    /* 1. process subsystem */
    tiku_htimer_init();     /* 2. hardware timer */
    tiku_timer_init();      /* 3. software timers */
}
Note: Initialization order is critical. The software timer subsystem starts a process (tiku_timer_process), so the process system must be initialized first. The hardware timer generates ticks that the software timer process consumes, so it must be initialized before software timers start waiting for ticks.

tiku_sched_loop()

Enters the main event loop, which never returns under normal operation:

  1. Autostart processes — starts all processes registered with TIKU_AUTOSTART_PROCESSES().
  2. Enable interrupts — calls __enable_interrupt() so ISRs (timer tick, UART RX, etc.) can fire.
  3. Main loop — drain events, then enter idle atomically.
void tiku_sched_loop(void)
{
#if TIKU_AUTOSTART_ENABLE
    tiku_autostart_start(tiku_autostart_processes);
#endif

    __enable_interrupt();

    while (sched_state == TIKU_SCHED_RUNNING) {

        /* Drain all pending work */
        while (tiku_sched_run_once()) {
            /* keep dispatching */
        }

        /* No more events -- enter idle atomically */
        tiku_atomic_enter();

        if (!tiku_sched_has_pending()) {
            if (idle_hook != (tiku_sched_idle_hook_t)0) {
                idle_hook();
            }
        }

        tiku_atomic_exit();
    }
}

API Reference

tiku_sched_run_once()

Dispatches one event from the process event queue. Timer polling is handled by the clock ISR (which calls tiku_timer_request_poll() on each tick) and by timer_insert() when a timer is set or reset.

uint8_t tiku_sched_run_once(void);
/* Returns 1 if an event was dispatched, 0 if idle */

tiku_sched_has_pending()

Checks if the timer linked list has any active entries. If timers are pending, the timer process will eventually generate events when polled. This is used in the idle path to decide whether to enter low-power mode.

uint8_t tiku_sched_has_pending(void);
/* Returns non-zero if work is pending */

tiku_sched_set_idle_hook()

Registers a function that is called whenever the scheduler has drained the event queue and no timers are immediately due. The typical use is to enter a platform-specific low-power mode (e.g., LPM0 on MSP430).

void tiku_sched_set_idle_hook(tiku_sched_idle_hook_t hook);
/* Pass NULL to clear the hook */
Note: The idle hook is called inside an atomic section. The platform's low-power instruction should atomically re-enable interrupts (the MSP430 HAL guarantees this).

tiku_sched_notify()

Called from ISR context (typically the clock tick ISR) to request the timer process to poll for expired timers. Internally this calls tiku_timer_request_poll(), which posts a TIKU_EVENT_POLL to the timer process.

void tiku_sched_notify(void);
/* Safe to call from any ISR */

tiku_sched_stop()

Sets a flag that causes tiku_sched_loop() to return on its next iteration. Primarily useful for test harnesses that need the scheduler to exit cleanly.

void tiku_sched_stop(void);
/* Sets sched_state = TIKU_SCHED_STOPPED */

The Atomic Idle Pattern

The idle section of the main loop uses tiku_atomic_enter() and tiku_atomic_exit() to prevent a race condition known as the "lost interrupt" problem:

/* Without atomic section (BROKEN):
 *
 *   if (!has_pending()) {   // <-- ISR fires here, posts event
 *       idle();             // <-- we sleep, missing the event
 *   }
 *
 * With atomic section (CORRECT):
 *
 *   tiku_atomic_enter();    // disable interrupts
 *   if (!has_pending()) {
 *       idle();             // LPM0 atomically re-enables interrupts
 *   }
 *   tiku_atomic_exit();     // restore interrupt state
 */

The key insight: the platform's low-power instruction (e.g., __bis_SR_register(LPM0_bits | GIE) on MSP430) atomically enters sleep and re-enables interrupts in a single instruction. This means no interrupt can fire between the "is there work?" check and the sleep entry, preventing lost wakeups.

Warning: If you implement a custom idle hook, your low-power entry must atomically re-enable interrupts. Simply calling __enable_interrupt() followed by a separate sleep instruction creates a window where an interrupt can be lost.

Scheduler Constants

Constant Value Description
TIKU_SCHED_RUNNING 0 Scheduler is running normally
TIKU_SCHED_STOPPED 1 Scheduler has been asked to stop
